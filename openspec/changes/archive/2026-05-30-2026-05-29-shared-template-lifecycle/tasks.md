> Extract onto it with NO behavior change, so Report instantiates the same lifecycle without forking.
> Every phase lands behind green tests; the Extract suite staying green is the bar. WIP cap = 3.
>
> **Execution: → SEQUENTIAL/TDD throughout.** This is a coupled foundation refactor — each phase
> changes a contract the next depends on, and the whole point is behavior-preservation on a live path
> (Extract). NOT a fan-out workflow. Failing-test-first each phase. The only ⟲ workflow-OK slice is
> the final call-site sweep IF the overlay rename touches many files against a fixed generic API.

## Phase 4 · Overlay generalization (app) — ⏸ DEFERRED to smart-report (2026-05-29)
**Decision:** the overlay generalization (`PendingSchemaOverlay`→`PendingTemplateOverlay`, the
`SchemaField*`→`TemplateItem*` family, generic ChatStore actions) renames the **live Extract editing
overlay** for a consumer that **does not exist yet** — the Report *builder* (smart-report S3a). Doing
it now is speculative generalization of working code ahead of its second consumer (anti-overengineering
posture). The generalization should be driven by the real requirements **when smart-report builds the
report builder** — deferral-to-point-of-need, NOT a fork (Report won't copy Extract's overlay; it'll
generalize at the moment of need). The `mergeOverlayForSave` casts that part of this referenced were
already removed in Phase 3 (typed `ExtractBody`). The foundation Extract uses now + smart-report's
*render* reads (types + persistence + route + read API) is complete (Phases 1–3 + 5 proof).
- [x] **Resolved within this change = decision made + MOVED to `smart-report` Phase 4** (its builder/S3a phase): generalize `PendingSchemaOverlay`→`PendingTemplateOverlay` + `SchemaField*`→`TemplateItem*` + generic ChatStore actions, driven by the report-builder's real requirements. *(The work itself is now smart-report's, tracked at its Phase 4 line; nothing for this change to do.)*

# Tasks — core data-model hardening (running fix list)

> Every item is tracked here so none reverts to a silent placeholder/hack. Each lands behind green
> tests, no user-visible regression. WIP cap = 3 in flight. `[D]` = a smart-report dependency.

> **Execution mode** — each `##` section is tagged with how to run it (decide at task-pickup):
> - **→ SEQUENTIAL/TDD** — coupled contract/foundation work: it changes a shared type/shape that other
>   tasks depend on, or it's a careful single-file refactor. Do these by hand, failing-test-first, one
>   at a time. Fanning them out means agents editing a moving target.
> - **⟲ WORKFLOW-OK (post-contract)** — independent-per-file fan-out that becomes safe *only after* its
>   contract has landed: applying a base/factory across N wrappers, isolated finding-fixes, authoring N
>   drift-guard tests. Each unit is verifiable alone. **Two hard prereqs before launching any workflow:**
>   (1) the contract/foundation it builds on is already merged; (2) the **middleware parallel-test flake
>   is fixed** (spawned task) — otherwise workflow verify stages throw false reds and the run is untrustworthy.
> - **◑ MIXED** — the base/factory is SEQUENTIAL; the application across call-sites is WORKFLOW-OK once
>   the base lands. The section notes the split.
>
> Rule of thumb: if two tasks would touch the same type or file, they're sequential. If each touches a
> different file against a *fixed* contract, they fan out.

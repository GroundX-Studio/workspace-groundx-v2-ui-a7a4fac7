# 2. Core SDK contribution

For the canonical contribution rules, read the repo's
[`AGENTS.md`](https://github.com/eyelevelai/groundx-python/blob/main/AGENTS.md).
The high-leverage sections are AGENTS.md §1 (setup), AGENTS.md §2
(generated-vs-hand-written), AGENTS.md §3 (contributing to generated code),
AGENTS.md §6 (credentials), AGENTS.md §7 (release flow), and AGENTS.md §8
(workflow). This reference summarizes the high-leverage points so an agent can
plan a contribution before opening the repo.

## 2.1 The generated-vs-hand-written boundary is `.fernignore`

Most of `eyelevelai/groundx-python` is **generated by Fern** from the upstream
GroundX API definition. The boundary between generated and hand-written code is
**`.fernignore`**, not the directory layout.

Files and directories listed in `.fernignore` survive regeneration; everything
else gets rewritten. Current `.fernignore` highlights (read the file directly
for the live list):

| Path | Why it's hand-written |
|---|---|
| `src/groundx/extract/` | The entire extract submodule. See [§3 extract](./03-extract.md). |
| `src/groundx/ingest.py` | Hand-written `client.ingest()` entry point wrapping the generated upload flow. |
| `src/groundx/csv_splitter.py` | Hand-written CSV-splitting utility used by ingest. |
| `README.md` | User-facing README — preserved across regenerations. |

A file's Fern-generated status is also detectable from its first non-blank line:
if it reads `# This file was auto-generated by Fern from our API Definition.`,
it is generated. But **`.fernignore` is authoritative** — use it.

## 2.2 `pyproject.toml` is regenerated; edit `.fern/metadata.json` instead

`pyproject.toml` is **not in `.fernignore`** — Fern regenerates it. Direct edits
to `pyproject.toml` get overwritten on the next regen.

To change deps, extras, or mypy config: edit `.fern/metadata.json` under
`generatorConfig`. The relevant keys:

- `extra_dependencies.<name>` — declare an optional dep with
  `{"version": "*", "optional": true}`.
- `extras.<name>` — append the dep name to the right extras list (e.g.
  `extras.extract` for the `groundx[extract]` group).
- `pyproject_toml` — string-of-TOML appended to the regenerated `pyproject.toml`;
  this is where existing mypy `[[tool.mypy.overrides]]` blocks live.

The repo's `AGENTS.md` §4 walks through the full pattern with examples.

## 2.3 Contributing to generated code

Generated-code changes happen **upstream**, not in this repo. There are two
upstream surfaces, with different intake paths:

- **API-shape changes** (new endpoints, fields, types) — edit
  `fern/openapi.yml` in the **`eyelevelai/eyelevel-fern-config`** repo (the
  Fern source of truth for the GroundX API definition). A maintainer then
  manually triggers the "Release Python SDK" workflow in that repo with a new
  version (`workflow_dispatch` → input the next `3.X.Y`). The workflow runs
  `fern generate` and pushes the regenerated SDK to `groundx-python` main as
  a "Release X.Y.Z" commit by `fern-api[bot]`.
- **Generator behavior changes** (how every generated Python file looks:
  client class shape, error types, serialization, retry policy) — submit a
  PR to the [Fern repository](https://github.com/fern-api/fern), where the
  Python SDK generator lives alongside the other SDK generators. This is a
  separate upstream from the GroundX-specific Fern config.

For a specific file in the generated tree that genuinely needs hand-written
behavior, add it to `.fernignore` and write it by hand. Use this **sparingly** —
every entry is a maintenance commitment that disconnects the file from upstream
generator improvements.

### How to coordinate an upstream API-spec change

The repo's `AGENTS.md` §3 says: external contributors open a GitHub issue on
`eyelevelai/groundx-python` describing the needed shape change; a maintainer
routes it to `eyelevelai/eyelevel-fern-config` and edits the OpenAPI definition
there. Internal contributors with access to `eyelevel-fern-config` can edit the
spec directly. Either way, the regenerated SDK arrives via the "Release Python
SDK" workflow.

### Why `.fernignore` matters operationally

The "Release Python SDK" workflow regenerates **every file** in `groundx-python`
not listed in `.fernignore`. If a file the team wants to keep is missing from
`.fernignore`, the next release wipes it. This is what happened to AGENTS.md
between Releases 3.5.3 and 3.5.4 — added in PR #4 but not in `.fernignore`,
and the next regen removed it. Always add a hand-written file to `.fernignore`
**before** the next release runs.

## 2.4 Hand-written tests for generated code

Hand-written tests that exercise generated SDK code live under `tests/custom/`
(per the repo's documented test convention, AGENTS.md §3). Everything else
under `tests/` (outside `tests/custom/` and `tests/extract/`) is regenerated
test scaffolding.

For tests of **hand-written** SDK code (the extract submodule), tests live under
`tests/extract/` — see [§3 extract](./03-extract.md).

## 2.5 Validation gates

`mypy` + `pytest` are the gates. Run before pushing:

```sh
poetry run mypy .
poetry run pytest -rP -n auto
```

CI runs the same plus the `aiohttp`-marked tests after installing the extract
extra:

```sh
poetry install --extras extract
poetry run pytest -rP -n auto
poetry install --extras extract --extras aiohttp
poetry run pytest -rP -n auto -m aiohttp
```

A `ruff` config exists in `pyproject.toml` but ruff is not currently a CI gate.
For files you've changed: `poetry run ruff check <paths>` and
`poetry run ruff format <paths>` (per repo `AGENTS.md` §8).

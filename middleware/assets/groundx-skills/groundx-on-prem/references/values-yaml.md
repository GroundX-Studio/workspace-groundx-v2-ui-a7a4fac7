# values.yaml Field Reference

This file is the **capability-grouped reference for the values.yaml fields a deployer actually touches**. It documents the fields, their defaults, when a deployer would change them, the gotchas, and the cross-field implications a deployer needs to reason about.

It is not an exhaustive every-field-the-chart-reads enumeration — chart-internal plumbing (busybox image, hard-coded ports, internal probe paths) is omitted. For the upstream source of truth, the canonical reference is the `groundx-on-prem` repo's `src/groundx/values.yaml`.

For inline annotated example values.yaml files, route to `examples/`. For the discovery questionnaire that drives values-authoring decisions, route to `references/values-authoring.md`. For the install ordering, route to `references/install-flow.md`.

## 1. How values.yaml is layered

A real-world deployment usually composes **multiple values files**, layered by `helm install -f file1 -f file2 …`. The conventional pattern in the upstream `groundx-on-prem` repo:

| File | Purpose | Checked into version control? |
| --- | --- | --- |
| `src/groundx/values.yaml` | Chart defaults — the upstream baseline. | Yes (in the chart). |
| `values.<env>.yaml` (e.g. `values.aks.yaml`, `values.extract.eks.yaml`) | Per-environment overrides — sizing, mode, engines, node-label scheme, deployment-mode toggles. | Yes (deployer's repo). |
| `values.<env>.secret.yaml` (e.g. `values.aks.secret.yaml`, `values.extract.eks.secret.yaml`) | Credentials — backing-service usernames / passwords / endpoints, GCP service account JSON, API keys for external LLMs. | **No.** Kept out of version control or stored encrypted. |
| `values.aws.local.yaml` + `storageclass/values.aws.local.yaml` | Generated by `terraform/aws/setup-eks` — captures the chosen pvClass + access mode + (when workspace enabled) the workspace PVC details. | Generated; conventionally gitignored. |

The split between `values.<env>.yaml` and `values.<env>.secret.yaml` is purely an operational convention — the chart treats them as one merged values document at install time. The convention exists so credentials can be kept out of version control while the rest of the configuration is reviewed.

For the per-file convention details, route to `references/credentials.md`.

## 2. Top-level fields

These fields live at the root of values.yaml.

### 2.1 `namespace`

- **Type:** string
- **Default:** `eyelevel`
- **Description:** Kubernetes namespace where all GroundX resources are created. Must match the namespace passed via `helm install -n <ns>` (Helm passes it explicitly; this field exists so chart-internal cross-references resolve to the same name).
- **When to change:** Set to a customer-specific or environment-specific name (`groundx-prod`, `groundx-stage`, etc.).
- **Gotchas:** The chart assumes it owns the namespace. Do not co-tenant GroundX with unrelated workloads. See `references/cluster-requirements.md` § 4.

### 2.2 `licenseKey`

- **Type:** string
- **Default:** `""` (empty — chart emits no admin/license block)
- **Description:** GroundX license key. The chart writes this into the application's runtime config when set.
- **When to change:** When your deployment requires a license key. Provided by EyeLevel / GroundX at procurement time.
- **Gotchas:** Treat as a credential — put in the secret companion file, not the main values file. If the chart's `admin:` block is otherwise empty and `licenseKey` is empty, the chart omits the entire admin config section.

### 2.3 `admin.*`

- **Type:** dict — `apiKey`, `email`, `password`, `username`, `imageRepository` — all strings, all optional.
- **Default:** `{}` (empty — no bootstrap admin user is created in the config).
- **Description:** Bootstrap admin credentials that the application reads at startup. When at least one credential is set, the chart emits an `admin:` block into the application's runtime config. `admin.imageRepository` (separate concern) overrides the image-repository prefix for admin-tooling container images (air-gapped / mirror deployments).
- **When to change:** Provide the first admin user's credentials (and/or a pre-issued API key) at chart-install time. After the application is up, additional admin / customer users are created through the application's own management API rather than through values.yaml. Override `admin.imageRepository` when the admin-tooling images live at a non-default registry path.
- **Gotchas:** Treat the credentials as secrets — secret companion file only, never the main values file. The fields are all optional individually; set the ones your bootstrap workflow needs.

### 2.4 `mode`

- **Type:** string — `"all"` (full platform, the default) or `"ingest"` (ingest-only).
- **Default:** `"all"` (when omitted from values.yaml).
- **Description:** Controls which subsystems are deployed. In `mode: ingest`, the chart deploys only the ingest-side microservices and does **not** require a retrieval DB (OpenSearch) — the chart skips deploying or wiring the search backend.
- **When to change:** Set to `ingest` when this deployment is upstream-only (ingest pipeline + handoff to a separate retrieval cluster) or when running a benchmarking / sizing run that doesn't serve search.
- **Cross-field implications:** When `mode: ingest`, the `search.*` block can remain at chart defaults (OpenSearch deploy can be skipped) — see `references/services-prereqs.md` § 1 and § 3.4.

### 2.5 `imageType`

- **Type:** string — `""` (default images) or `"chainguard"` (Chainguard hardened distroless).
- **Default:** `""`.
- **Description:** Selects between the default container images and the Chainguard-hardened distroless variants.
- **When to change:** Set to `chainguard` for FedRAMP / SOC2-stringent / security-compliance-focused deployments. See `references/image-variants.md` (planned).
- **Gotchas:** Chainguard images run as user `65532` (the chart automatically uses this UID when `imageType=chainguard`). The default images run as `1001`. Mixing image variants across pods is not supported.

### 2.6 `environment` *(internal-only, not in schema)*

- **Type:** string — `"prod"`, `"stage"`, `"dev"`.
- **Default:** `"prod"` (set by the chart's `groundx.environment` helper).
- **Description:** Application-level environment label written into the runtime config by the chart helper.
- **Status:** The chart's `groundx.environment` helper reads `.Values.environment` but the `values.schema.json` declares `additionalProperties: false` at the top level and does **not** list `environment`. Setting `environment:` in values.yaml will fail strict helm schema validation. Leave it unset and let the chart helper default it to `"prod"`. If a non-`prod` label is genuinely needed, the schema must be patched upstream first.

### 2.7 `serviceAccount.name`

- **Type:** string
- **Default:** `""` (chart-default ServiceAccount).
- **Description:** Name of an externally-managed Kubernetes ServiceAccount that chart-managed pods should attach. Used for workload identity — for example, IRSA on AWS EKS where the ServiceAccount is annotated with an IAM role ARN.
- **When to change:** When the deployment needs cloud-managed credentials (AWS S3 / SQS / RDS access via IRSA, Azure Workload Identity, GCP Workload Identity Federation) instead of static keys in values.yaml.
- **Cross-field implications:** Replaces static `file.username` / `file.password` credentials when the ServiceAccount is bound to an IAM role with the right permissions.

### 2.8 `engines.default`

Chart-level default LLM engine — sets the default model id, reasoning-effort tier, vision, plus per-engine connection / sizing / rate-limit knobs. Read by application components that delegate engine selection. Per-microservice overrides (e.g. `extract.agent.modelId`) take precedence.

- **`engines.default.engineId`** (string) — Model identifier (e.g. `gpt-5-mini`, `test-model`).
- **`engines.default.reasoningEffort`** (string) — `"low"`, `"medium"`, `"high"`.
- **`engines.default.vision`** (boolean) — `true` to enable vision-capable engine paths.
- **`engines.default.type`** (string) — Engine type passed through to the application's runtime config. Common values seen in upstream samples: `openai`, `azure`, `vllm`, etc. The chart does not enforce an enum here.
- **`engines.default.service`** (string) — Engine service identifier when multiple endpoints are configured.
- **`engines.default.baseUrl`** (string) — Engine endpoint URL (when the default endpoint must be overridden).
- **`engines.default.apiKey`** (string) — Outbound API key for the engine. Treat as a credential.
- **`engines.default.dataType`** (string) — Tensor data type (e.g. `bfloat16`, `bfloat8`).
- **`engines.default.maxInputTokens`** / **`engines.default.maxOutputTokens`** (int) — Token-window bounds.
- **`engines.default.maxRequests`** (int) — Max concurrent in-flight requests.
- **`engines.default.requestLimit`** (int) — Rate-limit threshold.

### 2.9 `busybox`

- **Type:** dict — `image`, `imagePullPolicy`.
- **Default:** chart-built default busybox image; `imagePullPolicy` follows `cluster.imagePullPolicy`.
- **Description:** Container image used by the chart's init containers (probe-and-wait dependencies, etc.). Air-gapped deployments override this to point at the mirrored image.
- **When to change:** Air-gapped deployments. Chainguard / FedRAMP-stringent deployments override to a Chainguard busybox variant.

### 2.10 `logLevel`

- **Type:** string — e.g. `"info"`, `"debug"`, `"warn"`, `"error"`.
- **Default:** chart-default (typically `info`).
- **Description:** Application log verbosity. Read into the runtime config and propagated to every chart-managed microservice.

### 2.11 `languages`

- **Type:** list of strings.
- **Default:** chart-default (typically `["en"]` or empty).
- **Description:** Languages the application is configured to support / advertise. Drives language-detection and routing in some pipeline stages.

### 2.12 `dashboard.serviceName`

- **Type:** string.
- **Default:** chart-default.
- **Description:** Service-name override for the dashboard frontend service.

### 2.13 `integration.search`

- **Type:** dict — `duration`, `fileId`, `modelId`.
- **Default:** `{}`.
- **Description:** Configuration for integration / smoke-test search behavior. Typically used during chart installation and verification, not at runtime.

### 2.14 `global`

- **Type:** dict (generic).
- **Default:** `{}`.
- **Description:** Generic hook for chart-wide global values. Read by subchart conventions but not interpreted by GroundX templates themselves. Most deployers leave this empty.

## 3. Cluster shape — `cluster:`

The `cluster:` block configures the underlying Kubernetes assumptions.

### 3.1 `cluster.pvClass`

- **Type:** string
- **Default:** `eyelevel-pv` (chart default, but typically overridden by the generated `values.aws.local.yaml`)
- **Description:** Storage class name used by chart-managed PVCs.
- **When to change:** Whenever the cluster's default StorageClass is not named `eyelevel-pv`. The `terraform/aws/setup-eks` script generates this field automatically based on the chosen EFS / EBS driver.

### 3.2 `cluster.pvAccessMode`

- **Type:** string — `ReadWriteOnce`, `ReadWriteMany`.
- **Default:** `ReadWriteMany` in chart defaults; reduced to `ReadWriteOnce` for block-volume storage classes.
- **Description:** Volume access mode requested by chart-managed PVCs.
- **When to change:** Set to `ReadWriteOnce` when the StorageClass only supports single-pod attachment (e.g. EBS without a multi-attach configuration). Use `ReadWriteMany` for EFS-backed deployments.
- **Cross-field implications:** Some chart features (workspace runner shared volume) prefer `ReadWriteMany`; check `references/workspace-architecture.md` (in `groundx-architecture`) for the implications.

### 3.3 `cluster.hpa`

- **Type:** boolean
- **Default:** `false`
- **Description:** Master switch for Horizontal Pod Autoscaler (HPA) resources across the chart. When `true`, the chart creates HPAs for every microservice that supports it. When `false`, replicas stay fixed at the configured `replicas.desired`.
- **When to change:** Set to `true` for production deployments where load varies. Leave `false` for benchmarking / dev / fixed-load.
- **Cross-field implications:** HPA requires the metrics microservice. When `cluster.hpa=true`, also set `metrics.enabled=true`. See `groundx-architecture/references/observability.md` § 5.1.

### 3.4 `cluster.type`

- **Type:** string — `"eks"`, `"aks"`, `"gke"`, `"openshift"`, `"minikube"`, or `""`.
- **Default:** `"eks"`.
- **Description:** Identifies the Kubernetes platform. The chart adjusts a small number of platform-specific behaviors based on this (e.g. symlink creation rules for OpenShift / minikube).
- **When to change:** Set per deployment platform.

### 3.5 `cluster.nodeLabels.*`

- **Type:** dict — `cpuMemory`, `cpuOnly`, `gpuLayout`, `gpuRanker`, `gpuSummary` — all strings.
- **Default:** `eyelevel-cpu-memory`, `eyelevel-cpu-only`, `eyelevel-gpu-layout`, `eyelevel-gpu-ranker`, `eyelevel-gpu-summary`.
- **Description:** The label *values* the chart's `nodeAffinity` and `tolerations` templates target. The chart looks for these values under the node-label key **`eyelevel_node`** on cluster nodes.
- **When to change:** Override to match an existing node-label scheme in the cluster.
- **Gotchas:** The label *key* (`eyelevel_node`) is not configurable through values.yaml — it is baked into the templates. Do not use the bare key `node` — it collides with reserved metric labels in Prometheus / Grafana. See `references/node-groups.md` § 1.
- **Cross-field implications:** A per-microservice `<microservice>.node:` field overrides the default selection per pod — see `references/node-groups.md` § 4.

### 3.6 `cluster.imagePullSecrets`

- **Type:** list of strings
- **Default:** `[]`.
- **Description:** Names of Kubernetes imagePullSecrets that every pod should attach for pulling container images.
- **When to change:** Always, when pulling from a private registry. The Chainguard image variant requires its own pull secret (typically `chainguard-pull-secret`); the upstream `groundx-on-prem` repo's example values for Chainguard demonstrates this.

### 3.7 `cluster.imagePullPolicy`

- **Type:** string — `"IfNotPresent"`, `"Always"`, `"Never"`.
- **Default:** `"IfNotPresent"`.
- **Description:** Default `imagePullPolicy` applied to all chart-managed pods.
- **When to change:** Set to `"Always"` when pinning to a tag that the deployer regularly re-pushes (e.g. `latest`, `chainguard`), or when CI / GitOps expects every reconcile to re-pull.

### 3.8 `cluster.secrets`

- **Type:** list of strings
- **Default:** `[]`.
- **Description:** Names of pre-existing Kubernetes Secrets in the GroundX namespace that the chart should reference via `envFrom` / volume mounts. The conventional pattern: install one Secret (e.g. `eyelevel-secret-credentials`) holding all credentials, and list it here so every pod gets the env vars it needs.
- **When to change:** When using the **Kubernetes-Secrets-reference** credential pattern instead of the secret-companion-file pattern. See § 8.

### 3.9 `cluster.preProcessors`

- **Type:** list of dicts — each entry `{processorId: int, type: string}`.
- **Default:** `[]`.
- **Description:** Registers extra pre-processors for the ingest pipeline. The `processorId` is an integer the application recognizes; the `type` is a known kind (e.g. `"extract"`).
- **When to change:** When enabling extraction and the deployment needs a pre-process handler bound to the extract pipeline.
- **Cross-field implications:** Typically paired with `extract.enabled: true`.

### 3.10 `cluster.validApiKeys`

- **Type:** list of strings (UUIDs).
- **Default:** `[]`.
- **Description:** API keys the chart bakes into the application's runtime config as recognized credentials. Treat as credentials — secret companion file or `cluster.secrets`-referenced Secret only.
- **When to change:** When bootstrapping a deployment with one or more known API keys (callback API keys, internal-service keys). Production deployments typically grow these through the application's management API rather than through values.yaml.

### 3.11 `cluster.hasMig`

- **Type:** boolean.
- **Default:** chart-default (typically `false`).
- **Description:** Signals that the cluster has NVIDIA Multi-Instance GPU (MIG) partitioning configured. When `true`, GPU microservices schedule against MIG-partitioned device profiles instead of whole GPUs.
- **When to change:** Set `true` for clusters running A100 / H100 GPUs partitioned via MIG to share physical cards across `layout-inference`, `ranker-inference`, and `summary-inference`.

### 3.12 `cluster.tls.existingSecret`

- **Type:** string.
- **Default:** `""`.
- **Description:** Name of a Kubernetes Secret containing TLS certs the chart should mount cluster-wide for in-cluster TLS. The Secret is expected to follow the `kubernetes.io/tls` shape (`tls.crt`, `tls.key`).
- **When to change:** When the cluster uses a custom CA / cert-manager-issued cert for in-cluster service-to-service TLS. Route to `references/tls-and-certs.md` for the full TLS authoring workflow.

## 4. Backing services

For the three-mode (existing customer-managed / operator-deployed-dedicated / cloud-managed) decision logic, route to `references/services-prereqs.md`. The field shapes below restate the canonical configuration surface for each.

### 4.1 `cache:` (Redis)

- **`cache.enabled`** (boolean, default `true`) — Master switch for the chart's bundled in-cluster Redis. Set to `false` when using `cache.existing.*`.
- **`cache.image`** (string) — Image override for the bundled Redis. Chainguard / FedRAMP deployments use this to point at `cgr.dev/eyelevel.ai/redis-fips:...`.
- **`cache.persistence.enabled`** (boolean) + **`cache.persistence.capacity`** (string, e.g. `8Gi`) — Optional persistence for the bundled Redis. When enabled, a PVC is created against `cluster.pvClass`.
- **`cache.existing.addr`** (string) — External Redis endpoint. Setting this supersedes the bundled Redis.
- **`cache.existing.port`** (int, default `6379`) — External Redis port.
- **`cache.existing.isCluster`** (boolean) — `true` when pointing at Redis Cluster mode.
- **`cache.existing.ssl`** (boolean) — `true` for TLS-encrypted external Redis.
- **`cache.metrics.enabled`** (boolean) — Redis metrics exporter for Prometheus.
- **`cache.metrics.image` / `cache.metrics.imagePullPolicy`** — Image override for the metrics-cache sidecar (independent of the main cache image).
- **`cache.serviceAccountName` / `cache.metrics.serviceAccountName`** (string) — Kubernetes pod-spec-style `serviceAccountName` (one word) for the bundled Redis / metrics-cache pods. Distinct from the top-level `serviceAccount.name` field — used to attach a per-backing-service workload identity.
- **`cache.metrics.existing.{addr, port, isCluster, ssl}`** — Point the metrics-cache at a *separate* external Redis endpoint, independent of the main cache.
- **`cache.resources.requests.cpu` / `memory`** — Resource requests for the in-cluster Redis pod when `cache.enabled=true`.

**When to change:** Set `cache.existing.addr` to point at AWS ElastiCache, Azure Cache for Redis, or any in-house Redis when you want to skip the bundled deploy. Use `cache.metrics.existing.*` separately when the deployer wants metric data on its own Redis.

### 4.2 `db:` (MySQL)

- **`db.enabled`** (boolean, default `true`) — Master switch for in-cluster Percona MySQL via the Percona operator.
- **`db.dbName`** (string, default `eyelevel`) — Database name used by the application.
- **`db.username` / `db.password`** (string) — Application MySQL user credentials. Treat as credentials (see § 8).
- **`db.privilegedUsername` / `db.privilegedPassword`** (string) — Privileged credentials used at chart-install time to create the application user, database, and tables. Treat as credentials.
- **`db.port`** (int, default `3306`) — Port for the bundled in-cluster Percona (distinct from `db.existing.port` for external endpoints).
- **`db.serviceName`** (string) — Service-name override for the bundled MySQL.
- **`db.customSQL`** (string, multi-line) — Custom SQL statements the chart runs at install time (e.g., extra users, tables, or grants).
- **`db.maxIdle`** (int, default `5`) — Application's max-idle connection pool size.
- **`db.maxOpen`** (int, default `10`) — Application's max-open connection pool size.
- **`db.rootCerts`** (string, multi-line PEM) — TLS root CA bundle used by the application to verify the bundled DB's certificate.
- **`db.existing.ro`** (string) — Read-only endpoint (replica). Supersedes the bundled Percona deploy.
- **`db.existing.rw`** (string) — Read-write endpoint (primary). Supersedes the bundled Percona deploy.
- **`db.existing.port`** (int, default `3306`).
- **`db.existing.rootCerts`** (string, multi-line PEM) — TLS root CA bundle for the external DB. Required when the external endpoint requires TLS (e.g., AWS RDS / Azure Database for MySQL with `require_secure_transport=ON`).

**When to change:** Set `db.existing.{ro,rw}` to point at AWS RDS, an in-house MySQL, or any operator-deployed MySQL outside this chart's scope. Provide `db.existing.rootCerts` when the external service requires TLS — see `values.aks.yaml` in the upstream repo for the canonical Azure DigiCert root CA example.

### 4.3 `file:` (Object Store)

- **`file.enabled`** (boolean, default `true`) — Master switch for in-cluster MinIO via the MinIO operator. Set `false` when using `file.existing.*`.
- **`file.bucketName`** (string, default `eyelevel`) — Object-store bucket / container name. Required in all modes.
- **`file.username` / `file.password`** (string) — Credentials for the bundled in-cluster MinIO. Treat as credentials.
- **`file.serviceType`** (string) — Optional at the top level (some sample files set it here when not using `existing`). For Mode 1 / Mode 3 the canonical placement is `file.existing.serviceType`.
- **`file.serviceName`** (string, default `minio`) — Kubernetes Service name for the bundled MinIO.
- **`file.port`** (int, default `9000`) — Port for the bundled MinIO.
- **`file.customDomain`** (string) — Override the in-cluster DNS name when callers need to reach the bundled MinIO via a custom domain.
- **`file.existing.serviceType`** (string) — `"s3"` or `"minio"`. Required when pointing at an external object store.
- **`file.existing.url`** (string) — External S3 / MinIO endpoint URL.
- **`file.existing.region`** (string) — Required when `serviceType: s3`.
- **`file.existing.port`** (int) — Optional, for non-default ports.
- **`file.token`** (string) — Token-based auth for the file storage backend (alternative to username/password for backends that support it).
- **`file.ingress.*`** (full ingress shape — see § 5.10 for the common ingress field list) — Exposes the bundled MinIO via an Ingress resource (rare; typically only useful for direct MinIO console access).

**Credentials.** `file.username` / `file.password` (top-level under `file:`, *not* under `file.existing.*`) carry credentials for both the bundled in-cluster MinIO and the external endpoint. For S3 these are the AWS access key id and secret access key (e.g., `values.cloud.yaml` sets `username: awsKey`, `password: awsSecret`). The chart can alternatively read AWS creds from environment variables (`AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, or the `GROUNDX_*` variants) when those are mounted via `cluster.secrets`, or from an IAM role bound to `serviceAccount.name` via IRSA.

**Canonical AWS-S3 block (from upstream sample values files):**

```yaml
file:
  bucketName: <bucket-name>
  existing:
    url: https://<bucket-name>.s3.<region>.amazonaws.com
    region: <region>
    serviceType: s3
```

### 4.4 `search:` (OpenSearch)

- **`search.enabled`** (boolean, default `true`) — Master switch for the bundled OpenSearch deploy.
- **`search.indexName`** (string, default `prod-1`) — Logical index name used by the application.
- **`search.port`** (int) — Port for the bundled OpenSearch (distinct from any port encoded in `search.existing.url`).
- **`search.serviceName`** (string) — Service-name override for the bundled OpenSearch.
- **`search.existing.url`** (string) — External OpenSearch endpoint. Supersedes the bundled deploy.
- **`search.username` / `search.password`** — Standard application credentials. Secret companion file only.
- **`search.privilegedUsername` / `search.privilegedPassword`** — Higher-privilege credentials used for index management. Secret companion file only.

**When to change:** Set `search.existing.url` to point at AWS OpenSearch managed or an in-house OpenSearch deployment.

**Cross-field implications:** When `mode: ingest`, the retrieval DB is not required — `search.*` can be left at chart defaults without consequence (see `references/services-prereqs.md` § 3.4).

### 4.5 `stream:` (Queue — Kafka or SQS)

- **`stream.enabled`** (boolean, default `true`) — Master switch for bundled in-cluster Kafka via Strimzi. Set `false` when using external Kafka or SQS.
- **`stream.key` / `stream.secret`** (string) — Default AWS access key id / secret access key used for all SQS topics that do not override per-topic. Treat as credentials.
- **`stream.token`** (string) — Default token-based auth used for all topics that do not override per-topic. Treat as a credential.
- **`stream.region`** (string) — Default AWS region for SQS topics.
- **`stream.port`** (int) — Port for the bundled Kafka cluster.
- **`stream.serviceName`** (string) — Service-name override for the bundled Kafka.
- **`stream.replicas`** (int) — Replica count for the bundled Kafka cluster (Strimzi).
- **`stream.retention`** (string / int) — Topic retention period (Strimzi default).
- **`stream.segment`** (string / int) — Topic segment size (Strimzi default).
- **`stream.existing.domain`** (string) — External Kafka domain. Supersedes the bundled Kafka.
- **`stream.existing.port`** (int, default `9092`).
- **`stream.topics.<topic-name>`** — Per-topic configuration. The canonical topic names are: `preProcess`, `process`, `summary`, `update`, `upload`. Per-topic fields:
  - **`type`** (string) — `"kafka"` or `"sqs"`.
  - When `type: kafka`: **`topic`** (string) — Kafka topic name. **`groupId`** (string) — Kafka consumer group id. **`broker`** (string, optional) — Override the Kafka broker for this topic only (otherwise inherits `stream.existing.domain` + `port`).
  - When `type: sqs`: **`url`** (string) — SQS queue URL. **`region`** (string) — AWS region for this SQS queue. **`key`** / **`secret`** (string, optional) — Per-topic AWS credentials override; falls back to `stream.key` / `stream.secret`.
  - **`token`** (string, optional) — Per-topic token-based auth override; falls back to `stream.token`.

**Visibility timeout (SQS):** SQS queues must have a visibility timeout of **10+ minutes** for all GroundX-bound topics. The chart does not enforce this — the deployer configures it on the queue itself.

**Cross-field implications:** SQS swaps a managed queue for the Kafka primitive. When using SQS, set `stream.enabled: false` and configure each topic individually with `type: sqs` and the appropriate `url`. The Kafka and SQS shapes are mutually exclusive per topic but the chart supports mixing across topics. For contributor/internal verification only, upstream sample values include all-SQS and mixed Kafka + SQS worked examples; do not surface customer-specific sample file names in user-facing output unless the user explicitly asks about upstream source files.

## 5. Application microservices

Every application microservice in the chart shares a common configuration pattern. The chart-managed backing-service pods (bundled `cache`, `cache.metrics`, in-cluster MinIO, etc.) accept the same fields under their own blocks. Under each microservice / backing-service block, the deployer-facing sub-fields are:

- **`enabled`** (boolean) — Master switch for the microservice. Default `true` for always-on microservices; `false` for opt-in microservices (extract, workspace).
- **`replicas.desired`** (int) — Replica count when HPA is off, or initial replica count when HPA is on.
- **`replicas.min`** / **`replicas.max`** (int) — HPA bounds, used only when `cluster.hpa=true`.
- **`replicas.{hpa, cooldown, gracePeriod, threshold, throughput}`** — Full per-microservice HPA configuration (used by `extract.*` sub-microservices and the workspace-runner sub-microservices). `hpa: true` enables autoscaling for this microservice independently of the global `cluster.hpa`. `cooldown` is the scale-down cooldown in seconds. `gracePeriod` is the pod-termination grace period. `threshold` is the per-replica throughput threshold that triggers scale-up. `throughput` is the per-replica target throughput.
- **`resources.requests.cpu` / `memory`** — Kubernetes resource requests.
- **`resources.limits.cpu` / `memory` / `nvidia.com/gpu`** — Kubernetes resource limits. GPU microservices set `nvidia.com/gpu: 1`.
- **`image`** (string) — Container image override. Default comes from the chart's image-repository scheme; per-microservice override is used for air-gapped / Chainguard / pinned-version deployments.
- **`imagePullPolicy`** (string) — Per-microservice override of `cluster.imagePullPolicy`.
- **`node`** (string) — Override the per-microservice node-label *value* (defaults come from `cluster.nodeLabels`).
- **`nodeSelector`** (dict) — Override the entire nodeSelector block for this microservice (used when the default `eyelevel_node`-key scheme doesn't fit).
- **`affinity`** / **`tolerations`** (dicts) — Full overrides for the chart's default `nodeAffinity` + `tolerations` blocks.
- **`annotations`** / **`labels`** (dicts) — Extra pod / Deployment metadata.
- **`containerPort`** (int) — Override the container port the microservice listens on.
- **`containerSecurityContext`** / **`securityContext`** (dicts) — Per-microservice pod / container security context.
- **`serviceAccount.name`** (string) — Per-microservice ServiceAccount override (separate from the top-level `serviceAccount.name`).
- **`serviceName`** (string) — Override the Kubernetes Service name for this microservice.
- **`workers`** / **`threads`** (int) — Per-microservice worker × thread counts (Python microservices).
- **`queue`** (string) / **`queueSize`** (int) — Microservice-specific queue name / queue-size tuning.
- **`timeout`** (int, seconds) / **`timeoutKeepAlive`** (int, seconds) — Request timeouts (API microservices).

The capability-group entries below call out the fields that are *unique* to each microservice rather than restating the common pattern.

### 5.1 `groundx:` (API + orchestration)

The main API entry point. Standard CPU microservice; stateless; replicas scale horizontally. Plus the following microservice-specific fields:

- **`groundx.serviceName`** (string, default `groundx`) — Kubernetes Service name. Override when running multiple GroundX instances in the same namespace (rare) or matching an external naming convention.
- **`groundx.serviceType`** (string) — Kubernetes Service type: `"ClusterIP"`, `"NodePort"`, `"LoadBalancer"`, or `"Route"` (OpenShift Route).
- **`groundx.isInternal`** (boolean) — When `serviceType: LoadBalancer`, signals whether the LB should be internal-only (cloud-specific annotation behavior).
- **`groundx.image`** (string) — Image override. Chainguard / air-gapped / pinned deployments use this.
- **`groundx.ingress.enabled`** (boolean) — When `true`, the chart creates an `Ingress` resource for the GroundX API.
- **`groundx.ingress.ingressClassName`** (string) — e.g., `alb` for AWS Load Balancer Controller, `nginx`, etc.
- **`groundx.ingress.annotations`** (dict) — Ingress-controller-specific annotations (AWS ALB annotations, NGINX rewrite rules, etc.).

### 5.2 `layout:` (Vision pipeline)

The layout pipeline is a sub-tree of microservices (`layout.api`, `layout.correct`, `layout.inference`, `layout.map`, `layout.ocr`, `layout.process`, `layout.save`), each with the common pattern. Plus `layout.serviceName` for service-name override and `layout.podMemory` for a pod-level memory budget override applied across the sub-tree.

- **`layout.api`** — CPU microservice. Exposes the layout API. Plus `layout.api.{ingress.* (common ingress shape, see § 5.10), serviceType, timeout, threads, workers}` for external exposure and request tuning.
- **`layout.inference`** — GPU microservice. Defaults to `nvidia.com/gpu: 1` requests and limits. ~2.5 GB GPU memory per worker+thread (chart default `1 × 6`). For GPU sizing details see `references/cluster-requirements.md` § 2.2 and `references/node-groups.md` § 3. Vision-model-specific fields:
  - **`deviceType`** (string) — `"gpu"` (default) or `"cpu"` for CPU fallback.
  - **`model.name`** (string) — Hugging Face model identifier (e.g., `google/gemma-3-4b-it`).
  - **`model.dataType`** (string) — `"bfloat16"`, `"bfloat8"`, etc.
  - **`model.maxInputTokens`** / **`model.maxOutputTokens`** / **`model.maxRequests`** / **`model.swapSpace`** (int) — vLLM-style tuning knobs.
  - **`pvc.{access, class, name, capacity}`** — Optional PVC for model-weight caching (avoid re-downloading on every pod restart).
  - **`queue`** (string) — Internal queue name override.
  - **`runtimeClassName`** (string) — Kubernetes `runtimeClass`, e.g., `"nvidia"` for the standard GPU Operator runtime, `"nvidia-container-runtime"` on AKS.
  - **`updateStrategy`** (string or dict) — Deployment update strategy. GPU microservices typically need `"Recreate"` rather than `"RollingUpdate"` because a single GPU cannot host two pods simultaneously.
- **`layout.ocr`** — CPU microservice (Tesseract). Switches to Google Cloud Vision when `layout.ocr.type: google` is set. The GCV credentials can be provided two ways:
  - **`layout.ocr.credentials`** (string) — Path to a GCP service-account JSON file *packaged in the chart* (e.g. `files/ocr/credentials.json`). At template time the chart calls `.Files.Glob` against this path; if the file exists, the chart materializes a `ConfigMap` named `<layout-service>-ocr-credentials-map` carrying the JSON as `credentials.json`. Mounted into the layout-ocr pod. **Not inline JSON**; the value is a path string the chart resolves from packaged files.
  - **`layout.ocr.project`** — GCP project id written into the runtime config when the GCV path is used.
  - See `groundx-architecture/references/layout-ocr.md` § 5.2.
- **`layout.process.batchSize`** (int) — Batch size for the layout-process worker.

### 5.3 `layoutWebhook:` / `preProcess:` / `process:` / `queue:` / `summaryClient:` / `upload:`

All standard CPU microservices on the orchestration tier. Common pattern applies. `preProcess` is typically the highest-memory CPU microservice — assign to `eyelevel-cpu-memory` node group via `preProcess.node: eyelevel-cpu-memory`.

### 5.4 `ranker:` (Re-ranker)

- **`ranker.api`** — CPU microservice. Plus `ranker.api.ingress.*` (common ingress shape, see § 5.10) when the ranker needs external exposure (rare; usually `ClusterIP` only), `ranker.api.serviceType`, and `ranker.api.timeout` (int, seconds) for request timeout tuning.
- **`ranker.inference`** — GPU microservice. ~1.25 GB GPU memory per worker+thread. Chart default `14 × 1` workers/threads, sized for a 24 GB GPU. Plus:
  - **`deviceType`** (string) — `"gpu"` (default) or `"cpu"` (unreliable; not production-supported).
  - **`pvc.{access, class, name, capacity}`** — Full PVC config for model-weight caching.
  - **`queue`** (string) — Internal queue name override.
  - **`updateStrategy`** (string or dict) — Deployment update strategy; typically `"Recreate"` for the same single-GPU reason as `layout.inference`.
- **`ranker.serviceName`** (string) — Service-name override.

### 5.5 `summary:` + `summaryClient:` (Summary stack)

Two flavors of summary execution:

**Flavor A — self-hosted summary stack** (default):
- `summary.api.enabled: true` — Service-side wrapper around the inference model.
- `summary.inference.enabled: true` — GPU inference. Gemma 3 model. ~12 GB GPU memory per worker+thread; 24 GB total sized for `1 × 1`.
- `summaryClient` calls the in-cluster `summary-api`.

`summary.api.*` (common pattern) plus:
- **`summary.api.serviceType`** (string) — Service exposure: `"ClusterIP"` (default), `"LoadBalancer"`, `"Route"` (OpenShift).
- **`summary.api.isInternal`** (boolean) — When `serviceType: LoadBalancer`, signals internal-only LB.
- **`summary.api.ingress.*`** (common ingress shape, see § 5.10) — When the summary API needs to be reachable externally (rare).
- **`summary.api.threads` / `workers` / `timeout`** — Worker / thread / request-timeout tuning.

`summary.inference.*` (common pattern) plus the vision-model-style fields:
- **`summary.inference.deviceUtilize`** (float 0–1.0, default `0.48`) — Fraction of GPU memory the inference pod is allowed to use. High-throughput deployments may raise this after GPU-capacity testing.
- **`summary.inference.deviceType`** (string) — `"gpu"` (default) or `"cpu"` (not architecturally supported for production).
- **`summary.inference.model.{name, dataType, maxInputTokens, maxOutputTokens, maxRequests, swapSpace}`** — vLLM-style tuning. `name` is the Hugging Face model id.
- **`summary.inference.model.kwargs`** (dict) — Engine-specific extra arguments passed through to the underlying inference server.
- **`summary.inference.model.reasoningEffort`** (string) — Per-microservice override of the chart-default reasoning effort.
- **`summary.inference.pvc.{access, class, name, capacity}`** — Optional PVC for model-weight caching.
- **`summary.inference.queue`** — Internal queue name override.
- **`summary.inference.runtimeClassName`** — Kubernetes `runtimeClass` for the GPU runtime.
- **`summary.inference.updateStrategy`** (string or dict) — Deployment update strategy; typically `"Recreate"` for single-GPU reasons.
- **`summary.defaultKitId`** (int) — Default "kit" id for summary generation (application-level configuration).
- **`summary.serviceName`** — Service name override.

**Flavor B — external LLM** (deployer opts out of self-hosted GPU):
- `summary.api.enabled: false` and `summary.inference.enabled: false`.
- **`summary.existing.serviceType`** (string) — Schema declares this as a free string. The chart's `groundx.summary.create` helper only special-cases three values (`"openai"`, `"openai-base64"`, `"azure"`) as automatic "skip self-hosted" triggers. Other values (`"deep-infra"`, custom names) are passed through to the application's runtime config but require `summary.existing.url` to be set in order to actually skip self-hosted. Default value is `"eyelevel"` (which functionally means "deploy self-hosted"). See `references/engines.md` § 1.
- **`summary.existing.url`** (string) — `/chat/completions` endpoint. Optional when `serviceType` is one of the three chart-recognized values; required for any other engine name (otherwise self-hosted stays enabled).
- **`summary.existing.apiKey`** (string) — Outbound API key. Treat as a credential.
- `summaryClient` calls the external endpoint instead.

**Cross-field implications:**
- When Flavor B is selected, the `eyelevel-gpu-summary` node group is not needed.
- Document content (preprocessed chunks) leaves the cluster on each summary call — trust-boundary crossing. See `groundx-architecture/references/summary-service.md` § 6 and `groundx-architecture/references/identity-and-trust.md` § 6.2.

### 5.6 `workspace:` (Workspace runner subsystem — opt-in)

- **`workspace.enabled`** (boolean, default `false`) — Master switch. When `false`, the workspace microservices are not deployed and GroundX Workspace facade endpoints return errors.
- **`workspace.image`** / **`workspace.imagePullPolicy`** — Workspace-wide image override applied to every sub-microservice that doesn't override.
- **`workspace.serviceName`** — Top-level service-name override.
- **`workspace.secretName`** — Alternative to `workspace.existingSecret`: a Secret name carrying workspace credentials (provider-flexible).
- **`workspace.token`** (string) — GitHub / GitLab token for outbound `git push`. Treat as a credential.
- **`workspace.existingSecret`** (string) — Alternative to `workspace.token`: a Kubernetes Secret name with key `WORKSPACE_RUNNER_TOKEN`.
- **`workspace.publishDryRun`** (boolean, default `true`) — When `true`, the publish pipeline runs through to "would push" without actually pushing. Flip to `false` to enable real pushes.
- **`workspace.gitProvider`** (string, default `github`) — `"github"` or `"gitlab"`.
- **`workspace.publishGithubWorkflowId`** (string, default `deploy.yml`) — Name of the GitHub Actions workflow file that the workspace publish step expects to trigger.
- **`workspace.managedRepoOwner`** (string) — GitHub org / GitLab group that owns the managed-scaffold repos (e.g., `GroundX-Studio`).
- **`workspace.managedRepoNamePrefix`** (string) — Prefix applied to managed-scaffold repo names at creation time.
- **`workspace.managedRepoVisibility`** (string) — `"private"`, `"public"`, `"internal"`.
- **`workspace.allowedCommands`** (string, comma-separated) — Whitelist of shell commands the workspace runner will execute (e.g., `"python,git"`).
- **`workspace.commandTimeoutSeconds`** (int) — Per-command timeout in the workspace runner.
- **`workspace.mysqlConnectTimeoutSeconds`** (int) — DB-connection timeout for the workspace runner.
- **`workspace.workspaceRoot`** (string, e.g., `"/mnt/workspaces"`) — Path inside the workspace runner pods where workspace data is mounted.
- **`workspace.workspaceMinFreeBytes`** / **`workspace.workspaceMinFreePercent`** (int) — Disk-space minimums for the workspace volume.
- **`workspace.celeryBrokerUrl`** / **`workspace.celeryResultBackend`** (string) — Celery broker / result-backend URLs (override the chart-default cache target). Used when the workspace runner shares Celery infrastructure with an external Redis.
- **`workspace.celeryGlobalKeyprefix`** (string) — Celery key prefix.
- **`workspace.celerySoftTimeLimitSeconds`** (int) — Celery soft time limit.
- **`workspace.celeryTaskAlwaysEager`** (boolean) — When `true`, Celery tasks execute synchronously (testing / single-tenant deployments).
- **`workspace.github.{apiBaseUrl, appId, installationId, privateKeyPem, privateKeySecret.{name,key}, secretName, tokenTtlSeconds}`** — GitHub App configuration when authenticating that way. Treat `privateKeyPem` as a credential. `apiBaseUrl` enables GitHub Enterprise. `secretName` is an alternative-shape Secret reference (carries the GitHub credentials in one Secret).
- **`workspace.gitlab.{apiBaseUrl, token, tokenSecret.{name,key}, secretName}`** — GitLab equivalents.
- **`workspace.pvc.{name, capacity, access, class}`** — PersistentVolumeClaim for the shared workspace data volume. `access: ReadWriteMany` and an EFS-style storage class strongly preferred for production.
- **`workspace.{api,provision,workspace,command,publish,cleanup}`** — Sub-microservices, each with the common pattern (replicas, resources, node, nodeSelector). Plus per-microservice `queue:` override and `replicas.throughput` / `replicas.hpa` for granular HPA tuning.
- **`workspace.api.{isInternal, serviceType, ingress.{enabled, hostName, ingressClassName, path}}`** — Exposure for the workspace API (external integrations need this reachable; internal-only deployments leave it `ClusterIP`).

**Cross-field implications:**
- Requires `workspace.token` OR `workspace.existingSecret` when enabled.
- The workspace runner is the only chart-managed microservice that emits outbound traffic to GitHub / GitLab. Network policy must allow egress to the configured git remote.
- For the architectural picture, route to `groundx-architecture/references/workspace-architecture.md`.

### 5.7 `extract:` (Extraction microservices — opt-in)

- **`extract.enabled`** (boolean, default `false`) — Master switch. When `false`, the extract microservices do not render at all.
- **`extract.serviceName`** (string) — Service-name override for the extract microservices.
- **`extract.cacheDirectory`** (string) — On-disk cache directory inside the extract pods.
- **`extract.callbackApiKey`** (string) — API key for callback URLs the extract pipeline POSTs to when extraction completes. Treat as a credential.
- **`extract.callbackUrl`** (string) — Default callback URL the extract pipeline POSTs results to (e.g., `https://api.groundx.ai/api`). Often overridden per workflow.
- **`extract.file.{bucketName, username, password, region, serviceType, url}`** — Extract-side override for the upload bucket. Used when the extract pipeline writes to a *different* S3 bucket than the main ingest pipeline. Same shape as `file.existing.*`. Treat `username` / `password` as credentials.
- **`extract.{agent,api,download,save}`** — Four sub-microservices. Each follows the common pattern (`enabled`, `replicas`, `resources`, `node`).

**`extract.agent`** specifics:
- **`apiKey`** (string) — Outbound LLM API key when `serviceType` is an external engine (`openai`, etc.). Treat as a credential.
- **`apiBaseUrl`** (string) — Endpoint override for the agent's outbound LLM calls (when the default for the chosen `serviceType` isn't right).
- **`serviceType`** (string) — Engine type: `"openai"`, etc.
- **`modelId`** (string) — Override the chart-default `engines.default.engineId`.
- **`model.reasoningEffort`** (string) — `"low"`, `"medium"`, `"high"`, or `null` to inherit `engines.default`.
- **`model.kwargs`** (dict) — Engine-specific extra arguments (e.g., `max_tokens: 2000`).
- **`queue`** (string) — Internal queue name override.
- **`existingSecret`** (boolean) + **`secretName`** (string) — When `existingSecret: true`, the extract-agent reads its credentials from the named Kubernetes Secret instead of inline `apiKey`.
- **`replicas.{desired, max, min, hpa, cooldown, gracePeriod, threshold, throughput}`** — Full HPA configuration. `cooldown` (int seconds) is the scale-down cooldown; `gracePeriod` (int seconds) is the termination grace period for in-flight extractions; `threshold` (int) is the per-replica throughput threshold that triggers scale-up; `throughput` (int) is the per-replica target throughput. `hpa: true` enables autoscaling for this microservice independently of `cluster.hpa`.

**`extract.api`** specifics:
- **`isInternal`** (boolean) + **`serviceType`** (string) — Exposure (`"ClusterIP"`, `"LoadBalancer"`, `"Route"`).
- **`ingress.{enabled, ingressClassName, annotations}`** — When the extract API is externally reachable.

**`extract.save`** specifics (Google Drive integration):
- **`driveId`** (string) — Google Drive folder / shared-drive ID where extraction results are written.
- **`templateId`** (string) — Google Sheets / Docs template ID used as the output template.
- **`gcpCredentials`** (string, JSON) — GCP service account JSON. Conventionally set via `helm install --set-file extract.save.gcpCredentials=/path/to/gcp.json` to keep the JSON out of values.yaml.
- **`existingSecret`** (boolean) + **`secretName`** (string) — Read GCP credentials from a Kubernetes Secret instead of inline.

**Cross-field implications:**
- Typically paired with `cluster.preProcessors: [{processorId: 13, type: extract}]` in the main `cluster:` block to register the extract pre-processor.
- Per-document extraction activation is further gated by the workflow API request carrying an extraction YAML — see `groundx-architecture/references/extraction-architecture.md`.

### 5.8 `metrics:` (Observability microservice)

`metrics` is a full common-pattern microservice (image, replicas, resources, node, nodeSelector, affinity, tolerations, labels, annotations, securityContext, containerSecurityContext, containerPort, serviceAccount.name, serviceName) plus:

- **`metrics.enabled`** (boolean, default `false`) — Master switch for the metrics microservice. **Required when `cluster.hpa=true`** — HPA scales every other pod against the signals this microservice publishes.
- **`metrics.serviceMonitor.enabled`** (boolean, default `false`) — Set to `true` when using the kube-prometheus-stack operator to register a `ServiceMonitor` for the chart's `/metrics` endpoint.
- **`metrics.useExisting`** (boolean) — When `true`, the chart skips deploying the metrics microservice and consumes metrics from an existing pre-installed metrics collector.
- **`metrics.image` / `metrics.imagePullPolicy`** — Image override.

**Cross-field implications:** Enabling HPA (`cluster.hpa: true`) without `metrics.enabled: true` results in autoscalers that have no signal to scale on. Use `metrics.useExisting: true` when the deployer already runs a metrics collector and doesn't want a duplicate from this chart.

### 5.9 `layoutWebhook:` (Webhook receiver — public-facing)

`layoutWebhook` is the only standard-tier microservice (besides `groundx`) that the chart can expose as a publicly-addressable Service. Common pattern plus:

- **`layoutWebhook.serviceType`** (string) — `"ClusterIP"`, `"LoadBalancer"`, `"Route"`, etc.
- **`layoutWebhook.serviceName`** (string) — Service-name override.
- **`layoutWebhook.ingress.*`** — Full ingress shape (see § 5.10).

When the chart is integrated with an asynchronous external pipeline (e.g., a managed extraction service that POSTs status callbacks), the external sender targets the layout-webhook ingress.

### 5.10 Common ingress shape

Every `<microservice>.ingress.*` block in the chart shares the same field set. Microservices with an ingress block: `groundx`, `layout.api`, `layoutWebhook`, `ranker.api`, `summary.api`, `extract.api`, `workspace.api`, `file`. Fields:

- **`enabled`** (boolean) — Master switch for creating the `Ingress` resource.
- **`ingressClassName`** (string) — e.g. `"alb"` (AWS Load Balancer Controller), `"nginx"`, `"azure-application-gateway"`.
- **`annotations`** (dict) — Ingress-controller-specific annotations (ALB scheme, NGINX rewrites, cert-manager-issuer, etc.).
- **`labels`** (dict) — Extra labels on the Ingress resource.
- **`hostName`** (string) — Hostname this Ingress responds for.
- **`path`** (string) — Single path for simple routing.
- **`paths`** (list of dicts) — Multi-path routing for non-trivial Ingresses.
- **`tls`** (list of dicts) — TLS config (Secret reference + hosts).
- **`apiVersion`** (string) — Override the `apiVersion` written into the Ingress resource (compat with older Kubernetes versions; chart picks a sensible default otherwise).

## 6. Throughput model — `throughput:`

The throughput model has two halves: **per-tpm budgets** (`throughput.tpm.*`) and **per-microservice throughput targets** (`throughput.services.*`). HPA scaling for every microservice references this block.

### 6.1 `throughput.tpm.*` — token-per-minute budgets

- **`throughput.tpm.document`** (int, default `12500`) — Tokens-per-minute budget per document.
- **`throughput.tpm.page`** (int, default `1000`) — Tokens-per-minute budget per page.
- **`throughput.tpm.extractRequest`** (int) — Tokens-per-minute budget per extraction request.
- **`throughput.tpm.summaryRequest`** (int) — Tokens-per-minute budget per summary request.

### 6.2 `throughput.services.*` — per-microservice throughput targets

Per-microservice throughput targets used by HPA to compute scale-up / scale-down. Each field is an integer in units appropriate to the microservice (pages-per-minute, queue-depth, etc.):

- **`throughput.services.layout.{correct, inference, map, ocr, process, save}`** — Per-stage layout pipeline targets.
- **`throughput.services.preProcess.queue`** — Pre-process worker queue-depth target.
- **`throughput.services.process.queue`** — Process worker queue-depth target.
- **`throughput.services.queue.queue`** — Queue worker target.
- **`throughput.services.ranker.inference`** — Ranker inference target.
- **`throughput.services.summary.{api, inference}`** — Summary API + inference targets.
- **`throughput.services.summaryClient.api`** — Summary client API target.
- **`throughput.services.upload.queue`** — Upload worker queue-depth target.

**When to change:** Tune to match the deployment's throughput targets. The values inform autoscaling targets — when HPA is on, pods scale based on observed throughput against these budgets. Production deployments typically tune `throughput.tpm.*` for the upstream LLM budget and tune `throughput.services.*` per-microservice to match observed bottlenecks under load.

## 8. Secrets: two patterns

The chart supports **two credential patterns**, often used together. Pick per credential based on the deployment's compliance posture.

### 8.1 Pattern A — inline in values.yaml

Set credential fields directly in the main values file (or, by convention, in a separate `values.<env>.secret.yaml` file kept out of version control and `-f`'d alongside the main file). The chart reads the values directly and writes them into the application's runtime config.

Fields that follow this pattern:

- `licenseKey` (top-level).
- `admin.{apiKey, email, password, username}`.
- `db.{username, password, privilegedUsername, privilegedPassword, rootCerts}` + `db.existing.{rootCerts}`.
- `file.{username, password}` (bundled MinIO) and `file.existing.{username, password}` (external).
- `search.{username, password, privilegedUsername, privilegedPassword}`.
- `stream.{key, secret}` and per-topic `stream.topics.<name>.{key, secret}` overrides.
- `summary.existing.apiKey`.
- `workspace.token`, `workspace.github.privateKeyPem`, `workspace.gitlab.token`.
- `extract.agent.apiKey`, `extract.callbackApiKey`, `extract.file.{username, password}`, `extract.save.gcpCredentials`.
- `cluster.validApiKeys` (list).

The `values.<env>.secret.yaml` companion-file convention is purely operational. The chart treats main + secret companion as one merged values document at install time. The split exists so credentials can be kept out of version control while the rest of the configuration is reviewed.

### 8.2 Pattern B — Kubernetes Secrets reference (production / stringent compliance)

Create a Kubernetes Secret out-of-band, then reference it from values.yaml. The application reads credentials as environment variables mounted from the Secret. This is the canonical pattern for production / SOC2 / FedRAMP deployments where credentials should never appear in any values file.

The upstream `groundx-on-prem` repo's **`groundx-secret` prereq chart** is the canonical way to create this Secret. Its values file has a *different shape* from the main chart — values.yaml-style fields are replaced with environment-variable-name keys under a `data:` block. Example structure (from `values.extract.eks.secret.yaml`):

```yaml
name: eyelevel-secret-credentials
namespace: eyelevel
data:
  GROUNDX_ACCESS_KEY_ID: <minio-or-aws-access-key>
  GROUNDX_SECRET_ACCESS_KEY: <minio-or-aws-secret>
  GROUNDX_ADMIN_EMAIL: <email>
  GROUNDX_ADMIN_API_KEY: <uuid>
  GROUNDX_ADMIN_PASSWORD: <password>
  GROUNDX_ADMIN_USERNAME: <uuid>
  GROUNDX_AGENT_API_KEY: <llm-api-key>
  GROUNDX_LICENSE_KEY: <uuid>
  MYSQL_INIT_PASSWORD: <pwd>
  MYSQL_INIT_USER: <root-user>
  MYSQL_USER: <app-user>
  MYSQL_USERNAME: <app-user>            # alias to MYSQL_USER on some deployments
  MYSQL_PASSWORD: <pwd>
  WORKSPACE_RUNNER_TOKEN: <github-or-gitlab-token>
  GITHUB_APP_PRIVATE_KEY_PEM: <private-key-pem>
```

After the Secret is installed (`helm upgrade --install groundx-secret groundx/groundx-secret -n eyelevel -f values.<env>.secret.yaml`), reference it from the main values.yaml so chart-managed pods mount it:

- **`cluster.secrets`** (list of strings, e.g., `["eyelevel-secret-credentials"]`) — Lists Kubernetes Secrets the chart wires into every pod via `envFrom`.
- **`workspace.existingSecret`** (string) — Names a Secret with `WORKSPACE_RUNNER_TOKEN` for the workspace runner.
- **`workspace.github.privateKeySecret.{name, key}`** — Names a Secret + key holding the GitHub App's private key.
- **`workspace.gitlab.tokenSecret.{name, key}`** — Names a Secret + key holding the GitLab API token.
- **`extract.agent.existingSecret: true` + `extract.agent.secretName: <name>`** — Names a Secret holding the extract-agent's LLM API key.
- **`extract.save.existingSecret: true` + `extract.save.secretName: <name>`** — Names a Secret holding the GCP service account JSON.

The application maps standard environment variable names (`GROUNDX_*`, `MYSQL_*`, `AWS_*`, `WORKSPACE_RUNNER_TOKEN`, etc.) to runtime configuration — so the same Secret can satisfy multiple credential needs at once.

### 8.3 Mixing patterns

A real deployment typically mixes both. Common patterns:

- **Inline DB creds + Secret-referenced workspace token + IRSA for S3.** Credentials with low sensitivity inline (or in a secret companion file), high-sensitivity tokens via Secret references, cloud credentials via workload identity (`serviceAccount.name` bound to an IAM role).
- **All-Secret-referenced.** Production / SOC2 / FedRAMP. No credentials in any values file; all wired via `cluster.secrets` + per-feature `existingSecret` fields.

The `groundx-storageclass` / `groundx-secret` / `groundx-service-account` **prereq charts** in `src/groundx/prereqs/` install the supporting Secret + ServiceAccount + StorageClass resources alongside the main `groundx` chart — see `references/install-flow.md` for the ordering.

The `groundx-storageclass` prereq chart accepts its own values shape (separate from the main chart's values.yaml). Example for EFS-backed EKS (from `values.extract.eks.sc.yaml`):

```yaml
storageClassName: eyelevel-pv
isDefaultClass: false
provisioner: efs.csi.aws.com
reclaimPolicy: Retain
parameters:
  provisioningMode: efs-ap
  fileSystemId: <fs-...>
  directoryPerms: "700"
  basePath: /eyelevel
```

For EBS-backed (default-class) deployments, swap `provisioner: ebs.csi.aws.com` + `reclaimPolicy: Delete` + `parameters: { type: gp3 }`. The `storageClassName` here must match `cluster.pvClass` in the main chart values.
